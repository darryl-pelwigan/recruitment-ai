# Recruitment AI

A full-stack recruitment platform that takes hiring from job posting to offer вЂ” with role-based dashboards, an AI-driven candidate scoring engine, a drag-and-drop hiring pipeline, and an automated documentation pipeline that generates a full user manual straight from the running app.

Built solo, end to end: database schema, API, auth, UI, and docs.

![Dashboard](docs/screenshots/dashboard.png)

---

## About This Project

Recruitment AI is a portfolio project that mirrors a real applicant-tracking system (ATS) like Greenhouse or Lever, scoped to what a small team would actually need. It's built around four roles вЂ” Applicant, Recruiter, HR, and Administrator вЂ” each with a distinct view of the same data, enforced by JWT auth and route guards on both the frontend and backend.

The goal was to demonstrate full ownership of a non-trivial product: a normalized relational schema with migrations, a clean service-layer API, a resume-matching engine written from scratch (no third-party AI API), a polished responsive UI, and documentation that stays in sync with the app because it's generated by scripting the app itself with Playwright.

---

## Highlights

- **Role-based access control** across 4 roles (Applicant / Recruiter / HR / Admin), enforced with JWT + protected routes on the frontend and dependency-injected auth checks on the backend.
- **Custom AI/NLP scoring engine** (`services/ai_service.py`) вЂ” parses uploaded resume PDFs, tokenizes them, and scores each application 0вЂ“100 against a job's required skills, description, and requirements, with no external LLM/API dependency.
- **Drag-and-drop hiring pipeline** вЂ” a Kanban-style board (Applied в†’ Under Review в†’ Shortlisted в†’ Interview Scheduled в†’ Rejected в†’ Hired) that updates application status on drop.
- **Rich text job descriptions** via a Tiptap-based editor, with file uploads for resumes, avatars, and company logos.
- **Analytics dashboard** вЂ” active jobs, applicant totals, conversion rate, top skills, and per-job applicant counts, tailored to the signed-in role.
- **Automated documentation pipeline** вЂ” Playwright drives the live app through every page and role, captures screenshots, and a script assembles them into a non-technical, numbered-procedure user manual exported to `.docx`. See [Documentation](#documentation).
- **Typed, validated forms end to end** вЂ” React Hook Form + Zod on the frontend, Pydantic schemas on the backend.

## Features

**For Applicants**
- Browse and search open jobs, filter by type, location, and posted date
- Apply with an auto-attached resume and cover message
- Track every application's status on My Applications
- Save jobs to revisit later
- Manage a profile resume plus a history of per-application resumes
- Edit profile, avatar, and password

**For Recruiters & HR**
- Post, edit, and close job listings with a rich text editor, salary range, and company branding
- Review applicants per job, including an AI-generated match score and cover message
- Move applicants through hiring stages via list view or a drag-and-drop pipeline board
- Search and filter the full applicant pool across all jobs, by skill or recent activity
- Bookmark promising applicants for future openings
- View dashboard analytics: active jobs, applicant volume, hires, conversion rate, top skills

**For Administrators**
- Everything Recruiters/HR can do
- Full user directory with search and role filtering
- Change any user's role or reset their password

## Getting Started

### Prerequisites

- Python 3.10+
- Node.js 18+
- PostgreSQL

### Backend Setup

```bash
cd backend
python -m venv venv

# Windows
venv\Scripts\activate

# macOS / Linux
source venv/bin/activate

pip install -r requirements.txt
```

Create a `.env` file inside `backend/`:

```env
DATABASE_URL=postgresql://user:pass@localhost/recruitment_db
SECRET_KEY=your-secret-key
ALGORITHM=HS256
ACCESS_TOKEN_EXPIRE_MINUTES=30
```

Run database migrations and start the API:

```bash
alembic upgrade head
uvicorn app.main:app --reload
```

- API runs at `http://localhost:8000/api`
- Interactive docs (Swagger UI) at `http://localhost:8000/docs`

### Frontend Setup

```bash
cd frontend
npm install
npm run dev
```

App runs at `http://localhost:5173`

### Environment Variables

| Variable | Description |
|---|---|
| `DATABASE_URL` | PostgreSQL connection string |
| `SECRET_KEY` | Secret key for JWT signing |
| `ALGORITHM` | JWT algorithm (default: `HS256`) |
| `ACCESS_TOKEN_EXPIRE_MINUTES` | Token expiry duration in minutes |

## Documentation

A full, non-technical **User Manual** covering every page and role is generated directly from the running application:

- [`docs/generated/UserManual.docx`](docs/generated/UserManual.docx) вЂ” the finished manual
- [`docs/markdown/`](docs/markdown/) вЂ” the markdown source, one file per page
- [`docs/screenshots/`](docs/screenshots/) вЂ” screenshots captured per page, per role

The process: Playwright logs in and navigates the live app page by page, capturing screenshots; each page is then written up with an overview, numbered procedure, notes, and tips; and the result is assembled into a formatted Word document вЂ” all driven through a set of project rules under [`.claude/rules/`](.claude/rules/) that keep terminology, structure, and screenshot standards consistent.

## Architecture Decisions

- **Routes are thin** вЂ” all business logic lives in the matching service in `services/`.
- **Schemas are separate from ORM models** вЂ” Pydantic schemas (`schemas/`) handle validation and serialization; SQLAlchemy models (`models/`) own persistence.
- **`ai_service.py` owns all NLP logic** вЂ” resume parsing, keyword extraction, and scoring are isolated from routes.
- **`security.py` is the single source of truth** for JWT encoding/decoding вЂ” token logic is never inlined elsewhere.
- **`config.py` is the single source of truth for settings** вЂ” env vars are read through one `Settings` object, not scattered `os.environ` calls.
- **Frontend API calls go through `src/api/api.ts`** вЂ” components and stores never call Axios directly.
- **Auth and theme state are centralized** in dedicated Zustand stores, not scattered `useState`.
